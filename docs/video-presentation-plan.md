# 🎬 План видео-презентации: «Развёртывание отказоустойчивой инфраструктуры MediaWiki в Yandex Cloud»

> **Длительность:** ~10 минут
> **Целевая аудитория:** Комиссия, преподаватели, коллеги-сисадмины
> **Формат:** Голос за кадром + демонстрация экрана (скриншоты / запись экрана)

---

## 0:00 – 1:00 | Вступление и цель проекта

**Что говорить:**

- Представиться, назвать тему: высокодоступная MediaWiki-инфраструктура
- Цель: продемонстрировать навыки проектирования и развёртывания инфраструктуры для ~40 пользователей
- Ключевой принцип: Infrastructure as Code — Terraform + Ansible
- Бюджет: ~924 руб/мес за счёт прерываемых ВМ и HDD (экономия ~70%)

**Что показывать на экране:**

- Диаграмму архитектуры (`docs/diagrams/architecture.drawio` — экспортировать в PNG)

## 1:00 – 3:00 | Архитектура инфраструктуры

**Что говорить:**

- 7 серверов в 5 изолированных подсетях (VPC `10.0.0.0/16`)
- **Три уровня:**
  - **Веб-уровень:** LB-01 (Nginx, единственный с публичным IP)
  - **Прикладной уровень:** APP-01 + APP-02 (MediaWiki 1.43 + PHP-FPM)
  - **Уровень данных:** DB-01 (PostgreSQL Master) + DB-02 (Streaming Replica)
- **Вспомогательные сервисы:** ZABBIX-01 (мониторинг), BACKUP-01 (бэкапы)
- Все серверы — Ubuntu 22.04 LTS, прерываемые ВМ

**Что показывать на экране:**

- Ту же диаграмму, но с акцентом на потоки трафика:
  - Пользователь → LB → APP → DB
  - Пользователь → LB → /zabbix → ZABBIX
  - BACKUP → DB (pg_dump) + APP (rsync)

---

## 3:00 – 4:30 | Процесс развёртывания (общий обзор)

**Что говорить — три этапа, без показа кода:**

1. **Terraform** — создание облачной инфраструктуры:
   - VPC с 5 подсетями
   - 5 security groups с правилами наименьших привилегий
   - 7 виртуальных машин с нужными характеристиками

2. **Скрипт update-inventory.sh** — автоматическая синхронизация IP-адресов из Terraform в Ansible-инвентарь

3. **Ansible** — настройка серверов:
   - 5 ролей: nginx-lb, mediawiki, postgresql, zabbix, backup
   - Один playbook (`site.yml`) разворачивает всё за один запуск

**Что показывать на экране:**

- Схему-блок: `Terraform → update-inventory → Ansible → Готово`

## 4:30 – 6:30 | Демонстрация работающего продукта

**Что показывать на экране (скриншоты или запись экрана):**

1. **Открыть MediaWiki** — `http://<LB_IP>`
   - Главная страница вики
   - Создать тестовую страницу, показать что работает

2. **Открыть Zabbix** — `http://<LB_IP>/zabbix`
   - Dashboard с хостами и метриками
   - Показать статусы серверов

3. **Проверить бэкапы** — показать листинг `/backup/mediawiki/db/` и `/backup/mediawiki/fs/`
   - Файлы `.sql.gz` и `.tar.gz` с датами

**Что говорить:**

- MediaWiki доступна через балансировщик
- Zabbix мониторит все 7 серверов
- Бэкапы создаются автоматически по расписанию (БД в 02:00, ФС в 03:00)

---

## 6:30 – 8:00 | Высокая доступность и отказоустойчивость

**Что говорить:**

- **Балансировка:** Nginx upstream с `max_fails=3 fail_timeout=30s` — при падении одного APP-сервера трафик автоматически идёт на второй
- **Репликация БД:** PostgreSQL streaming replication (Master → Replica). При сбое мастера — ручное переключение на реплику (`pg_promote()`)
- **Бэкапы:** Ежедневные, хранение 7 дней, на изолированном сервере
- **План восстановления:** Документирован в `docs/recovery-plan.md` — сценарии сбоя каждого компонента

**Что показывать на экране:**

- Фрагмент `recovery-plan.md` — таблицу с симптомами и временем восстановления
- Или схему failover: DB-01 ✗ → DB-02 становится master

---

## 8:00 – 9:00 | Безопасность

**Что говорить:**

- **Единственная точка входа:** Только LB-01 имеет публичный IP
- **SSH jump-host:** Все внутренние серверы доступны только через ProxyJump
- **Security Groups:** 5 групп с минимальными правилами:
  - APP принимает HTTP только из lb-subnet
  - DB принимает PostgreSQL только из app-subnet
  - Бэкапы — только из db-subnet и app-subnet
- **SSH по ключам**, без паролей

**Что показывать на экране:**

- Краткую таблицу security groups (какая подсеть → какая подсеть)
